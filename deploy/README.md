# Deploy ‚Äî operator handbook

This folder holds the container infrastructure for Mail Aggregator.
The architecture itself is normative in `docs/07-deployment.md`; this
README is operational ‚Äî what to run, in what order, with what flags.

## Files

| Path | Purpose |
| --- | --- |
| `Dockerfile` | Multi-stage build, two final targets: `api` and `worker`. |
| `nginx/nginx.conf` | Main nginx config (mounted read-only at `/etc/nginx/nginx.conf`). |
| `nginx/templates/default.conf.template` | Per-host server block; rendered at container start by the alpine entrypoint, which `envsubst`'s `${SERVER_DOMAIN}`. |

For the operator-side bootstrap (DNS, firewall, first cert, GH secrets), see
`docs/SERVER-SETUP.md` ‚Äî it's the canonical first-run runbook.

The compose file lives at the repo root (`/docker-compose.yml`) so
`docker compose up` works without `-f`.

## Dependency-manager contract (READ ME, backend agent)

The Dockerfile assumes `uv` (Astral) for Python dependencies. **Backend
agent must produce, at the repo root**:

1. `pyproject.toml` ‚Äî declares `[project]`, `[project.dependencies]`,
   and (optionally) `[project.optional-dependencies] dev = [...]` for
   ruff / mypy / pytest.
2. `uv.lock` ‚Äî committed. Generated by `uv lock`.

The build runs `uv sync --frozen --no-dev --no-install-project` first
(deps only, max layer cache) then a second pass with the source code.
If the lock is stale, the build fails fast ‚Äî that's intentional.

If the backend agent picks `pip` + `requirements.txt` instead, the
Dockerfile's `builder` stage needs editing (replace the two `uv sync`
lines with `pip install --prefix=/opt/venv -r requirements.txt`).
We chose `uv` because it's 10‚Äì100√ó faster in CI and the lock file
makes builds deterministic.

## First boot (developer machine)

```bash
# 1. Generate secrets
cp .env.example .env
# Edit .env ‚Äî at minimum set:
#   POSTGRES_PASSWORD
#   MAIL_ENCRYPTION_KEY  (python -c "import os, base64; print(base64.b64encode(os.urandom(32)).decode())")
#   ADMIN_PASSWORD

# 2. Build images and start the stack
docker compose up -d --build

# 3. Watch logs
docker compose logs -f api worker

# 4. Confirm health
docker compose ps           # every service should show "(healthy)"
curl http://localhost:8080/healthz   # only works if you publish api port (we don't by default)
```

In dev the api port is **not** published to the host (the bundled
`docker-compose.override.yml` re-publishes `127.0.0.1:8080:8080`). To hit
it without the nginx front (which only runs under `--profile prod`),
either:

- exec into the network: `docker compose exec api curl localhost:8080/healthz`
- or temporarily add `ports: ["8080:8080"]` under `services.api` in a
  `docker-compose.override.yml` (gitignored ‚Äî your local convenience).

## Upgrade

```bash
git pull
docker compose build api worker
docker compose up -d api worker     # postgres / redis untouched
```

Migrations run via a dedicated **`mas-migrations`** init container
(`alembic upgrade head`, exits 0). Both `api` and `worker` `depends_on`
its `service_completed_successfully`, so neither boots against a
schema-less DB. The api image still ships `alembic` as a CLI ‚Äî the init
container just invokes it. `seed_super_admin` remains a startup hook
inside `api` (runs after migrations because of the dependency chain).

Migration policy: forward-only, online (no table-locking ALTERs).
Hot-fix migrations also forward-only ‚Äî never `alembic downgrade` in prod.

## Rollback

```bash
git checkout <previous-tag>
docker compose build api worker
docker compose up -d api worker
```

If a migration was applied that the previous code can't read, write a
forward-fix migration; do not `downgrade`.

## Backups (host cron, not in this repo)

```bash
# Postgres ‚Äî daily 02:00, retain 14 days
docker exec mas-postgres pg_dump -U mas -d mail_aggregator -F c \
  | gzip > /backups/pg/$(date +%F).dump.gz
```

Both must be encrypted (gpg) before going off-site.
**`MAIL_ENCRYPTION_KEY` lives in a separate vault** ‚Äî without it the
postgres dump is useless (mail-account passwords un-decryptable).

## Troubleshooting

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| `api` stuck "starting" forever | DB migrations failing | `docker compose logs api` ‚Äî look for alembic stack trace. |
| `worker` healthcheck flaps | `/tmp/worker_alive` not written | Check worker process logs ‚Äî APScheduler keep-alive job may have crashed. |
| 503 from any endpoint | postgres or redis unhealthy | `docker compose ps`; restart the unhealthy service. |
| TLS cert renewal failed | certbot can't reach LE / port 80 blocked | `docker compose logs certbot`; check UFW on 80; verify DNS A record matches host IP. |
| nginx serves stale TLS cert after renewal | nginx not reloaded since cert rotated | `docker compose exec nginx nginx -s reload` (or rely on the weekly host cron ‚Äî see `docs/SERVER-SETUP.md` Part D). |
| `502 Bad Gateway` from nginx | api unhealthy | `docker logs mas-api`. |
