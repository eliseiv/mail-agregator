# ADR-0007: Схема хранения вложений в MinIO

- **Статус:** accepted
- **Дата:** 2026-05-05

## Context

Вложения писем — бинарные данные произвольного размера. Хранить в PostgreSQL (BYTEA / large objects) — плохо: раздувает БД, ломает бэкапы, мешает репликации. Естественный выбор — object storage; ТЗ зафиксировал MinIO.

Нужно определить:
- Имя bucket'а.
- Схему ключей (чтобы безопасно расшаривать, легко чистить и не пересекаться между пользователями).
- Жизненный цикл (retention).

## Decision

- Bucket: один — `mail-attachments`. Создаётся автоматически при первом старте API (idempotent `head_bucket`/`create_bucket`).
- Bucket policy: **private**. Объекты не публикуются. Скачивание — только через backend (presigned URL короткой длительности или streaming).
- Схема ключей:
  ```
  {user_id}/{mail_account_id}/{message_uid}/{attachment_id}_{safe_filename}
  ```
  где:
  - `user_id`, `mail_account_id`, `message_uid` — числовые/uuid идентификаторы из БД;
  - `attachment_id` — внутренний BIGINT id из таблицы `attachments`, гарантирует уникальность даже если в письме два файла с одинаковым именем;
  - `safe_filename` — оригинальное имя, прошедшее санитайзер (`re.sub(r'[^A-Za-z0-9._-]', '_', name)`, длина <= 100).
- Content-Type сохраняется как метадата объекта.
- Размер вложения ограничен 25 MiB (см. `06-security.md`); при превышении вложение не сохраняется, в БД помечается `skipped_too_large=true`.
- Скачивание клиентом — через эндпойнт `GET /api/messages/{id}/attachments/{aid}`, который проверяет владение и стримит из MinIO.
- Удаление: при удалении пользователя / аккаунта / письма — каскадно удаляются объекты MinIO. Для письма cleanup-job при retention делает `delete_objects` пачкой по 1000.

## Consequences

**Плюсы:**
- Чёткая иерархия упрощает каскадное удаление по префиксу (`{user_id}/`, `{user_id}/{account_id}/` и т.д.).
- Один bucket — проще управление policy.
- attachment_id в имени снимает коллизии.

**Минусы:**
- Удаление по префиксу в S3 API — это `list + delete batch`, не атомарно. Mitigation: cleanup-job устойчив к повторному запуску (idempotent), может оставлять "потерянные" объекты — раз в неделю запускается `orphan_scan` (опционально, см. tech-debt registry — на текущей итерации не реализуем).

## Alternatives considered

- **Локальный диск с volume**: проще, но vendor-lock на конкретный путь; сложно мигрировать в облако.
- **Хранение в БД (BYTEA)**: ломает бэкапы, увеличивает размер in-memory кэша Postgres.
- **Несколько bucket'ов на пользователя**: усложняет management; нет выгоды.
