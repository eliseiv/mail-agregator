# ADR-0011: Ретенция писем и вложений — 30 дней

- **Статус:** accepted
- **Дата:** 2026-05-05

## Context

ТЗ требует чистить старые письма раз в сутки. Цели:
- Ограничить рост БД и MinIO.
- Не хранить личные данные пользователей дольше необходимого.
- Сервис — agregator/inbox, а не архив; для архива пользователи используют свой почтовый сервер.

## Decision

- TTL для писем (`messages`) и связанных вложений (`attachments` + объекты в MinIO) — **30 дней** с момента `internal_date` письма.
- Поле для базы расчёта: `messages.internal_date` (= `INTERNALDATE` IMAP, дата получения провайдером).
- Cleanup job: запускается worker'ом ежедневно в 03:00 UTC.
- Алгоритм:
  1. SELECT id, mail_account_id, user_id FROM messages WHERE internal_date < now() - INTERVAL '30 days' LIMIT 5000.
  2. Для каждой батч-партиции:
     - SELECT id, s3_key FROM attachments WHERE message_id = ANY(:ids).
     - DELETE объектов из MinIO батчами по 1000 через `delete_objects`.
     - DELETE FROM attachments WHERE message_id = ANY(:ids).
     - DELETE FROM messages WHERE id = ANY(:ids).
  3. Повторять, пока не вернёт 0 строк.
- **Sent messages** хранятся **бессрочно** (это собственные данные пользователя; объём минимальный). Это **намеренное design-решение**, а не tech-debt: 30-дневная ретенция сознательно не применяется к таблицам `sent_messages` / `sent_attachments`. Если в будущем потребуется TTL — отдельный ADR.
- Удаление пользователя / аккаунта — каскадное немедленное удаление, не ждёт cleanup job.

## Consequences

**Плюсы:**
- Простая, предсказуемая политика.
- Размер БД ограничен предсказуемо (30 дней × ~5 пользователей × ~100 ящиков × среднее ~50 писем/день ~= 750 000 строк max).

**Минусы:**
- Пользователь не может найти старое письмо в сервисе — должен идти в свой родной webmail. Это сознательное ограничение.

## Alternatives considered

- **Бессрочное хранение**: при росте — вырастет в проблему backups/disk.
- **Разная ретенция per user / per account**: лишняя гибкость, не запрошено.

## Revisions

- **2026-05-05 (rev. 2):** явно зафиксировано, что бессрочное хранение `sent_messages` / `sent_attachments` — намеренное design-решение, а не tech-debt. Соответствующий пункт в реестре tech-debt помечен как отозванный (см. `100-known-tech-debt.md`, закрытый ~~TD-009~~).
