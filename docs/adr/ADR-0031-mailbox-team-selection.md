## Context

`mail_accounts.group_id` существует с ADR-0019 и определяет принадлежность ящика **одной** команде (1:1). По текущей реализации (`backend/app/accounts/service.py:340-395`) команда ящика жёстко присваивается при создании как **домашняя группа владельца** (`owner.group_id`), и сменить её после создания нельзя — поля выбора команды в форме нет, а edit-форма явно объявляет «переназначить владельца/команду нельзя» (`08-frontend.md` §4.7).

Это создаёт две проблемы для пользователей, состоящих в нескольких командах (ADR-0030):

1. **Нельзя выбрать команду при добавлении.** Пользователь-член нескольких команд не может положить новый ящик в неосновную команду — он всегда попадает в домашнюю.
2. **Нельзя перенести существующий ящик** между командами, в которых он виден, без удаления и пересоздания (что теряет кэш писем).

Продукт **запросил** обе возможности. Решения пользователя, зафиксированные как вход для этого ADR:
- Делаем **И** селектор команды при добавлении, **И** перенос существующего ящика.
- `group_member` видит в селекторе **только свои** команды (его `user_groups`), по умолчанию — домашняя (`users.group_id`); в чужую команду положить нельзя.

`mail_accounts.group_id` **уже** есть в схеме — миграция БД не требуется. Дубликаты по email в разных командах остаются раздельными строками (round-18, `find_by_user_email` / `list_canonical_account_ids`).

---

## Decision

### 1. `mail_accounts.group_id` выбирается при создании и может меняться; миграция не нужна

`mail_accounts.group_id` остаётся **1:1** (ящик принадлежит ровно одной команде — ADR-0030 §3 неизменен). Меняется только то, что эта команда теперь **выбирается** (а не выводится из `owner.group_id` принудительно) и **может быть изменена** после создания. Схема БД не меняется — колонка уже nullable BIGINT FK (`03-data-model.md` таблица `mail_accounts`).

### 2. Добавление с выбором команды

`MailAccountCreateRequest` получает опциональное поле `group_id: int | None`.

- **Если `group_id` НЕ передан (None)** — поведение как сейчас: ящик попадает в **домашнюю** группу владельца (`owner.group_id`); для `super_admin`, создающего на себя (`target_user_id` = self / опущен), домашней группы нет → `mail_accounts.group_id = NULL` (персональный ящик). Полная обратная совместимость.
- **Если `group_id` передан** — backend валидирует допустимость целевой команды по роли создателя (см. §4) и сохраняет ящик в указанную команду.

Резолв владельца (`target_user_id`) по ADR-0019 §8 **не меняется**; `group_id` — ортогональное поле, валидируется отдельно и **после** резолва владельца.

### 3. Перенос существующего ящика — расширение `PATCH /api/mail-accounts/{id}`

Перенос реализуется **расширением существующего** `PATCH /api/mail-accounts/{id}` опциональным полем `group_id: int | None`, а **не** отдельным sub-resource. Обоснование:
- Согласуется со стилем method-override/form-fallback: путь `^/api/mail-accounts/\d+$` **уже** в whitelist (`_OVERRIDE_REGEX_PATHS`) для `PATCH` через `POST … + _method=PATCH`. **Новый regex-путь не добавляется** → счётчик `_OVERRIDE_REGEX_PATHS = 16` в `tests/unit/test_method_override.py` **не меняется**, `docs/04-api-contracts.md` §8 form-fallback-таблица для accounts **не получает новой строки**.
- Перенос команды — это изменение поля существующего ресурса, семантически это PATCH, не отдельная сущность.

`MailAccountUpdateRequest` получает поле `group_id: int | None` плюс sentinel `set_group_id: bool` (по образцу `clear_display_name`), чтобы отличить «поле не передано» (не менять команду) от «передано» (сменить, в т.ч. на `NULL` для super_admin-персонального). В JSON: присутствие ключа `group_id` ⇒ менять; в form-fallback: присутствие form-поля `group_id` ⇒ менять (пустое значение ⇒ `NULL`, только для super_admin).

Перенос **не требует** повторного IMAP/SMTP-теста (credentials и хосты не трогаются — меняется только принадлежность команде). Если в том же PATCH меняются хосты/порты/пароль — тест выполняется по существующим правилам ADR-0019/§accounts (`05-modules.md` §9).

Нужен repo-метод `MailAccountsRepo.update_group(account_id, group_id)` + проверка прав в `MailAccountService.update`. Дубликаты по email в разных командах остаются раздельными (round-18) — перенос не схлопывает и не конфликтует с `UNIQUE(user_id, email)` (он на `(user_id, email)`, а не на команде).

### 4. Правила допустимости целевой команды (единые для create и transfer)

Целевая команда `group_id` валидируется одинаково при создании (§2) и переносе (§3). Цель должна существовать (`groups.id`), иначе `404 group_not_found`.

| Роль инициатора | Допустимая целевая команда |
| --- | --- |
| `group_member` | `group_id` ∈ **собственные** `user_groups` (членства инициатора, = `scope.group_ids`). Только на себя (`target_user_id` = self по ADR-0019 §8). Иное → `403 user_not_in_group_scope`. |
| `group_leader`, создавая/перенося **себе** | `group_id` ∈ его `user_groups` (`scope.group_ids` — домашняя + доп. членства). Иное → `403 user_not_in_group_scope`. |
| `group_leader`, создавая/перенося **участнику своей команды** | `group_id` = **команда лидера** (его домашняя, `scope.group_id` = `groups.leader_user_id` группа). Положить ящик участника в другую команду лидер не может. Иное → `403 user_not_in_group_scope`. |
| `super_admin` | **любая** существующая `group_id` ИЛИ `NULL` (персональный ящик без команды). |

Нарушение → `ValidationError` (`400`) / `ForbiddenError` (`403 user_not_in_group_scope`) / `NotFoundError` (`404 group_not_found`) — **никогда `500`**.

`group_member` **не может менять команду существующего ящика** (transfer запрещён для `group_member`): даже если ящик ему виден через членство, изменение принадлежности ящика команде — административно-значимое действие, и для `group_member` оно недоступно. Попытка передать `group_id` в `PATCH` со стороны `group_member` → `403 forbidden`. (Создание с выбором своей команды `group_member`-у разрешено — это его собственный новый ящик; перенос уже существующего — нет.)

`group_leader` может переносить ящики **в пределах своей видимости** (ящики его команд) только в допустимые целевые команды по таблице выше. `super_admin` — любой ящик в любую команду/`NULL`.

Целевая команда переноса валидируется так же, как при создании (таблица выше). Дополнительно для transfer: backend проверяет, что **сам ящик** виден инициатору (`get_for_group_or_owner` / `get_for_scope`), иначе `404` (как для любого PATCH сегодня).

### 5. Источник списка команд для селектора — `GET /api/my/groups`

Вводится новый лёгкий endpoint `GET /api/my/groups`, возвращающий команды, доступные текущему пользователю как целевые для селектора:
- `group_member` / `group_leader` — список его команд из `scope.group_ids` (`[{id, name}]`, отсортирован по `name`).
- `super_admin` — **все** существующие группы (flat-метод `GroupsRepo.list_all_groups()` — не paginated, в отличие от admin-списка `list_paginated`), плюс на фронте опция «Без команды» (`group_id=null`).

Endpoint **переиспользует** `GroupsRepo` (`05-modules.md` §6 groups) — не дублирует логику. Не вводится новый сервис: тонкая обёртка читает `scope.group_ids` → `GroupsRepo.list_by_ids` (для не-admin) или `GroupsRepo.list_all_groups()` (для super_admin — flat-метод). Это легче, чем прокидывать список в контекст каждой страницы, и переиспользуется JSON-клиентом и form-страницей одинаково.

`super_admin`/лидер уже имеют `GET /api/admin/groups`, но он admin-only (super_admin) и тяжелее (members_count и пр.); `GET /api/my/groups` — user-scope, доступен всем ролям, отдаёт минимум (`id`, `name`).

### 6. Audit

Перенос существующего ящика пишет `admin_audit` действие **`mail_account_group_change`** с `details = {mail_account_id, from_group_id, to_group_id}`. Пишется при изменении `group_id` через PATCH — для `super_admin` (всегда) и для `group_leader` (как обычное admin-significant действие на ящик). Создание ящика с выбором команды **не** пишет отдельного audit — это часть обычного create-flow (как сегодня create не пишет audit).

> **Замечание о консистентности с §1.7/§9.** Действия `group_leader`/`group_member` обычно не пишутся в `admin_audit` (это user-actions, ADR-0019 §9). Перенос команды ящика — исключение: это редкое структурное изменение принадлежности данных, поэтому фиксируется для super_admin'ского аудит-следа. `actor_user_id` = инициатор переноса, `target_user_id` = владелец ящика (`mail_accounts.user_id`).

## Consequences

### Положительные
- **Без миграции БД.** `mail_accounts.group_id` уже есть; меняется только логика присвоения и добавляется UPDATE.
- **Без нового form-fallback-пути.** Перенос идёт через существующий `PATCH /api/mail-accounts/{id}` → `_OVERRIDE_REGEX_PATHS` (16) и тест не меняются.
- **Единые правила авторизации** для create и transfer — одна функция валидации целевой команды, без дублирования.
- **Обратная совместимость create.** `group_id` опционален; без него — текущее поведение (домашняя группа / NULL для super_admin self).
- **Переиспользование `GroupsRepo`** — без дублирования групп-логики.

### Отрицательные / компромиссы
- **`mail_accounts.group_id` теперь mutable.** Видимость ящика для участников меняется при переносе. Это by design; ревокация сессий не требуется (видимость считается на каждом запросе из `mail_accounts.group_id` ∈ `scope.group_ids`, а не кэшируется в сессии — в отличие от `users.group_id`, который кэшируется и потому требует revoke при смене членства, ADR-0030).
- **Дубликаты ящика по email.** При переносе ящик с email X в команду, где уже есть другой ящик с email X (другого владельца), остаётся отдельной строкой — два ящика одного email в одной команде. Это согласовано с round-18 (дубликаты раздельны; super_admin inbox дедупит через `list_canonical_account_ids`). Дополнительной блокировки не вводим — сценарий редкий и не ломает консистентность.
- **Транзитивная видимость при переносе.** После переноса ящик исчезает из видимости тех, кто видел его только по старой команде, и появляется у новой. Поскольку видимость не кэшируется, эффект мгновенный — приемлемо.

## Alternatives considered

1. **Отдельный sub-resource `POST /api/mail-accounts/{id}/transfer` (или `/group`).** Отвергнуто. Потребовал бы **новый** regex-путь в `_OVERRIDE_REGEX_PATHS` (+ бамп счётчика теста до 17, новая строка в form-fallback-таблице) и новый роут-хендлер ради семантики, которая полностью укладывается в PATCH одного поля. Расширение PATCH проще и консистентнее с edit display_name/хостов.

2. **Сделать `mail_accounts.group_id` множественным (M:N ящик↔команда).** Отвергнуто (как ADR-0030 §3 / A2). Ломает 1NF, конфликтует с canonical-дедупом, требует пересмотра sync и шифрования. Принадлежность ящика остаётся 1:1; «видеть ящики других команд» решается членством зрителя, а не дублированием ящика.

3. **Переиспользовать `GET /api/admin/groups` для селектора вместо нового endpoint.** Отвергнуто. Он super_admin-only (`group_member`/`group_leader` получат 403) и отдаёт тяжёлый payload (members_count, leader). Лёгкий user-scope `GET /api/my/groups` нужен всем ролям.

4. **Прокинуть список команд в template-context страницы вместо JSON-endpoint.** Отвергнуто. JSON-клиент (AJAX-форма) всё равно нуждается в endpoint; дублировать источник между контекстом и API — рассинхрон. Один endpoint обслуживает оба (no-JS страница тоже может отрендерить `<option>`-ы из того же сервиса при server-render формы, но источник истины — один сервис-метод).

5. **Разрешить `group_member` переносить существующий ящик в свои команды.** Отвергнуто (по решению пользователя). `group_member` может выбрать команду только при **создании** своего ящика; перенос уже существующего — административно-значимое действие, доступное `group_leader` (в его scope) и `super_admin`. Это явно зафиксировано, чтобы не оставлять дыру в спецификации авторизации.
