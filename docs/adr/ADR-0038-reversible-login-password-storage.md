# ADR-0038 — Обратимое хранение пароля входа оператора для показа админом

| | |
| --- | --- |
| Статус | accepted |
| Дата | 2026-07-06 |
| Заменяет / отменён | — (**дополняет** ADR-0006 «argon2id для паролей входа» — хеш остаётся источником истины для проверки входа; **переиспользует** AES-256-GCM примитив ADR-0005) |

## Context

Внутренний инструмент; единственный `super_admin` (оператор) администрирует небольшой набор операторов-пользователей (`group_leader`/`group_member`, ≤ 5–25 человек, ADR-0019/ADR-0030). Возникло эксплуатационное требование: **на странице `/admin` super_admin должен видеть пароль входа оператора** (колонка «Пароль»), чтобы передать/подсказать его сотруднику, не инициируя полный reset-флоу с самостоятельной установкой пароля.

**Проблема.** Пароль входа сейчас хранится **только** необратимо — `users.password_hash` (argon2id, ADR-0006). Из хеша исходный пароль восстановить нельзя by design. Значит, «показать пароль» без изменения схемы хранения невозможно.

Ограничения и требования:
- **Argon2-хеш остаётся источником истины для проверки входа** (ADR-0006 не отменяется). Обратимая копия — **дополнительная**, только для показа; на путь аутентификации не влияет.
- Существующие операторы (у кого пароль задан до этой фичи и не менялся) не имеют обратимой копии → колонка показывает «—», пока пароль не будет заново задан админом или самим пользователем.
- Не плодить новые секреты/ключи — переиспользовать существующий `MAIL_ENCRYPTION_KEY` (ADR-0005).
- Показ — только `super_admin`; каждый показ — в audit; значение пароля — **никогда** в логи.
- Осознанный security-компромисс: это внутренний инструмент, оператор доверенный; риск (доступ к БД+ключу раскрывает пароли входа) принят явно (см. Consequences).

## Decision

Хранить пароль входа оператора **дополнительно** в обратимо-зашифрованном виде (AES-256-GCM тем же `MAIL_ENCRYPTION_KEY`), рядом с необратимым argon2-хешем. Админ **задаёт** пароль при создании/сбросе оператора; в этот момент пишутся и хеш, и зашифрованная копия. Показ — по требованию через отдельный super_admin-endpoint с расшифровкой, rate-limit и audit.

### 1. Модель + миграция

Новая колонка `users.password_encrypted BYTEA NULL` (SQLAlchemy `Mapped[bytes | None]`, `LargeBinary`, nullable):
- **NOT NULL** — обратимая копия пароля есть (задан админом или установлен пользователем после запуска фичи).
- **NULL** — обратимой копии нет (пароль предшествует фиче и не менялся, либо сброшен в self-set-режиме без задания пароля админом) → UI-колонка «Пароль» показывает «—».

Миграция Alembic **`20260706_022_user_password_encrypted`**, `down_revision='20260703_021'` (последняя — ADR-0034 mail-forwarding). Forward-only: `ADD COLUMN password_encrypted BYTEA NULL`. Backfill **не** делается (старые записи остаются NULL — «—»). Крипто-схема (§2) не требует предсказания id (в отличие от `mail_accounts`): `user_id` уже известен на момент шифрования (create — после INSERT users; reset/self-set — user существует).

### 2. Шифрование (переиспользование AES-256-GCM)

Новые helper'ы в `shared/crypto.py` — `encrypt_user_password(plaintext, user_id) -> bytes` / `decrypt_user_password(blob, user_id) -> str` — тот же примитив AES-256-GCM и тот же `MAIL_ENCRYPTION_KEY`, что у почтовых паролей (ADR-0005), но с **отдельным AAD-доменом**:

```
key  = base64decode(env.MAIL_ENCRYPTION_KEY)      # тот же 32-byte ключ
iv   = os.urandom(12)                              # 96 бит, per-record
aad  = b"user_pw:" + str(user_id).encode("ascii")  # domain-separation + биндинг к user_id
ct   = AESGCM(key).encrypt(iv, plaintext.encode("utf-8"), aad)
blob = b"\x01" || iv (12B) || ct_with_tag          # version_byte как у ADR-0005 (rotation)
```

**Domain separation через AAD.** Префикс `b"user_pw:"` отличается от `b"mail_account_password|"` (ADR-0005 §2) и `b"webhook_secret|"` (ADR-0023 §2.1). Атакующий с доступом к БД **не** может подставить blob из `mail_accounts.encrypted_password`/`webhooks.secret_encrypted` в `users.password_encrypted` — расшифровка упадёт на `InvalidTag`. Биндинг к `user_id` блокирует перестановку между пользователями (blob оператора A не расшифруется под AAD оператора B).

**Ротация `MAIL_ENCRYPTION_KEY`** (`mas-cli/worker reencrypt`, ADR-0005 §10 / `06-security.md` §10): `users.password_encrypted` добавляется в список обрабатываемых blob-колонок наравне с mail-паролями, webhook-секретами и OAuth-токенами (общий `version_byte`: 0x00 = старый ключ, 0x01 = новый).

### 3. Точки записи обратимой копии (везде, где известен plaintext)

| Путь | Где | Что пишем |
| --- | --- | --- |
| Пользователь сам задаёт пароль | `auth.complete_set_password` (self-set после reset/seed) | и argon2-хеш, и `password_encrypted` — колонка остаётся корректной после самостоятельной смены пароля (без рассинхрона). |
| Админ создаёт оператора с паролем | `admin.create_user`, опциональное поле `password` | argon2-хеш + `password_encrypted`, `password_reset_required=false`. |
| Админ сбрасывает пароль оператору с паролем | `admin.reset_password`, опциональное поле `password` | argon2-хеш + `password_encrypted`, `password_reset_required=false`. |
| Админ создаёт/сбрасывает **без** пароля | те же методы, поле `password` пусто | прежний self-set-флоу: `password_hash=NULL` (или force self-set), `password_reset_required=true`, `password_encrypted=NULL` → колонка «—». Полная обратная совместимость. |

**НЕ делаем** (осознанно, по решению с пользователем): backfill обратимой копии на успешном логине (когда plaintext доступен в момент проверки). Это оставило бы окно, где старый пароль «—» до первого входа; принято не усложнять — старые пароли показываются как «—» до явного задания админом/пользователем.

### 4. Endpoint показа

**`GET /api/admin/users/{id}/password`** (super_admin only):
- Расшифровывает `password_encrypted` под AAD этого `user_id`, возвращает `{"password": "<plaintext>"}`.
- `password_encrypted IS NULL` → `404 password_not_set` (нет обратимой копии — «—»).
- Не-super_admin → `403 forbidden`. Несуществующий user → `404 not_found`.
- **Read-only, НЕ мутация** → метод `GET`, **method-override не применяется** (см. §7).
- **Rate-limit** `LIMIT_ADMIN_PASSWORD_REVEAL` (env `ADMIN_PASSWORD_REVEAL_RATE_LIMIT_PER_MINUTE`, `int`, default `30`, `ge=1`) — **per-actor** (по `super_admin` `user_id`), анти-bulk-exfiltration: ограничивает массовую выкачку всех паролей скриптом и не даёт заспамить audit.
- **Audit** `user_password_revealed` на **каждый** успешный показ (`actor_user_id=super_admin`, `target_user_id={id}`, `details={}` — **без значения пароля**).
- Значение пароля **не логируется** ни в structlog, ни в audit.details; response этого endpoint исключается из request/response-логирования (только статус-код).

Расширение существующих endpoint'ов полем `password` (JSON + form-fallback):
- **`POST /api/admin/users`** (create) — `password?: str` (12..128, те же правила силы, что self-set: ≥1 буква + ≥1 цифра). Задан → admin-set-флоу (§3); пуст/отсутствует → self-set-флоу.
- **`POST /api/admin/users/{id}/reset`** — `password?: str` (те же правила). Задан → admin-set-флоу; пуст/отсутствует → текущий force-self-set-флоу.
- Audit `user_password_set` при задании пароля админом (`actor=super_admin`, `target_user_id={id}`, `details={}` — без значения). Пишется **в дополнение** к `create_user`/`reset_password`.

### 5. Мульти-команды при создании (сопутствующее, опирается на ADR-0030)

Отдельного ADR не требует — это применение готовой M:N-модели `user_groups` (ADR-0030). `CreateUserRequest` получает опциональное `additional_group_ids: list[int]`; `admin.create_user` после создания пользователя и **домашнего** членства (`user_groups` для `group_id`) вставляет членства `user_groups` для каждой из доп. команд в **той же транзакции**. Правила: только для `role='group_member'` (лидер привязан к своей команде; super_admin — без команд, `NULL`); дедуп с домашней `group_id` и между собой (`INSERT ... ON CONFLICT (user_id, group_id) DO NOTHING`); валидация существования команды (несуществующая → `400`). Переиспользует `UserGroupsRepo` (тот же код, что `POST /api/admin/users/{id}/groups`). См. `04-api-contracts.md` `POST /api/admin/users` и ADR-0030.

### 6. UI (кратко; детали — `08-frontend.md`)

- Колонка/поле **«Пароль»** в строке пользователя на `/admin`, между «Имя» и «Роль», **только для super_admin**. `has_password=true` (= `password_encrypted IS NOT NULL`, добавляется в `UserDTO`) → маска `••••••` + кнопка-«глаз»; клик → `fetch GET /api/admin/users/{id}/password` on-demand, показать значение, повторный клик — скрыть. `has_password=false` → «—». Пароль **не** встраивается в DOM заранее (грузится по требованию — минимальная экспозиция + audit на каждый показ).
- Диалоги create/reset — опциональное поле ввода пароля (админ задаёт).
- Мультивыбор доп. команд в диалоге создания (чекбоксы/multi-select из `groups`), только для `group_member`.

### 7. Взаимодействие с method-override (ADR-0015)

- **`GET /api/admin/users/{id}/password`** — `GET`, **не мутация** → `_method`-override не нужен; **новый override-путь не вводится**.
- Поле `password` в create/reset использует **существующие** whitelisted-пути (`/api/admin/users` exact; `^/api/admin/users/\d+/reset$` regex) — новых путей нет.
- `additional_group_ids` в create использует тот же существующий путь `/api/admin/users`.
- **Итог: `_OVERRIDE_EXACT_PATHS` остаётся 7, `_OVERRIDE_REGEX_PATHS` остаётся 16.** Хардкод-счётчики в `tests/unit/test_method_override.py` (`TestRegexCount`: exact == 7, regex == 16) **менять не нужно**.

## Consequences

**Положительные:**
- super_admin видит пароль входа оператора без reset-флоу — закрывает эксплуатационное требование.
- Argon2 остаётся источником истины входа; обратимая копия изолирована (отдельная колонка, отдельный AAD-домен, не участвует в проверке пароля).
- Ноль новых секретов/ключей — переиспользован `MAIL_ENCRYPTION_KEY` и существующий крипто-примитив; ротация ключа покрывает новую колонку тем же механизмом.
- Обратная совместимость: старые пароли → «—»; поле `password` в create/reset опционально — существующий self-set-флоу не меняется.
- Нет миграции данных (backfill), нет предсказания id, нет нового rate-limit-инфраструктурного кода (тот же slowapi-паттерн).

**Отрицательные / компромиссы (принято явно):**
- **Обратимость снижает конфиденциальность пароля входа.** Кто получил одновременно дамп БД (колонку `password_encrypted`) **и** `MAIL_ENCRYPTION_KEY`, — восстановит пароли входа операторов в открытом виде. Ранее (только argon2) это было невозможно. Это **осознанный компромисс для внутреннего инструмента**.
  - Меры снижения: (1) показ и endpoint — **только super_admin**; (2) **каждый показ** → audit `user_password_revealed`; (3) значение пароля **никогда не логируется** (ни structlog, ни audit.details, ни access-log); (4) `MAIL_ENCRYPTION_KEY` — env-only, `chmod 600`, не в git/логах (ADR-0005); (5) per-actor rate-limit против массовой выкачки; (6) пароль грузится в UI по требованию, не встраивается в разметку списка.
  - Граница риска не расширяется относительно уже принятого: тем же ключом уже обратимо шифруются **почтовые** пароли и OAuth-токены (ADR-0005/ADR-0025) — компрометация «БД+ключ» и так раскрывала бы их. Новая колонка добавляет к этому множеству пароли входа операторов.
- Пароль входа теперь существует в БД в (обратимо) восстановимой форме — при экспорте/бэкапе БД он присутствует. Бэкапы БД и `MAIL_ENCRYPTION_KEY` должны храниться раздельно (уже требование `06-security.md` §11).
- Небольшой рост строки `users` (blob ~30–50 байт на пользователя; объём ничтожен, ≤ 25 пользователей).

## Alternatives considered

1. **Не хранить пароль вовсе, показывать нечего (статус-кво argon2-only).** Отклонено: не удовлетворяет требование «показать пароль». Argon2 необратим by design.
2. **Не хранить, а генерировать одноразовый пароль при сбросе и показать его один раз (one-shot reveal, как webhook-secret ADR-0023).** Отклонено для основного сценария: требование — видеть **текущий** пароль оператора в любой момент из списка `/admin`, а не только в момент генерации. One-shot не даёт повторного показа; оператор, забывший пароль, снова требовал бы reset. (Тем не менее для новых операторов admin-set + показ фактически даёт эквивалент «показать при создании» **плюс** повторный показ.)
3. **Хранить пароль plaintext в БД.** Отклонено: даже для внутреннего инструмента plaintext-хранение — неоправданный шаг вниз. Дамп БД **без** ключа тогда сразу раскрывал бы все пароли; AES-GCM с `MAIL_ENCRYPTION_KEY` поднимает планку (нужен ещё и ключ, хранящийся отдельно) при том же удобстве показа. Выбран **AES-encrypted**.
4. **Отдельный ключ `LOGIN_PASSWORD_ENCRYPTION_KEY` вместо `MAIL_ENCRYPTION_KEY`.** Отклонено как усложнение: новый секрет, отдельная ротация, отдельный env — без выигрыша в модели угроз (оба ключа жили бы в одном `.env` на одном сервере; компрометация env раскрывает оба). Domain-separation достигается **AAD-префиксом** `user_pw:` в рамках общего ключа — достаточно и проще.
5. **Хранить обратимую копию для ВСЕХ (backfill на логине).** Отклонено (решение с пользователем): усложняет флоу и расширяет окно хранения без запроса; старые пароли остаются «—» до явного задания. При необходимости — отдельный ADR.

## Open questions

- **Q-0038-1** (закрыт в рамках этого ADR): нужен ли backfill старых паролей на логине? — **Нет**, по решению с пользователем (Alternative 5). Старые → «—».
