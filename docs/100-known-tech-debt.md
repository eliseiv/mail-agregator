# 100. Known Tech Debt

Реестр осознанных компромиссов и пунктов, отложенных на будущие итерации. Каждый item имеет ID `TD-NNN`, контекст, влияние и приемлемость.

| ID | Краткое название | Контекст | Impact | Severity | Когда адресовать |
| --- | --- | --- | --- | --- | --- |
| **TD-003** | Нет Prometheus-метрик | На первой итерации только structlog JSON в stdout. Подсчёт SLI — grep по логам. | Низкий: scope маленький. | low | Когда понадобится дашборд / алертинг. |
| **TD-004** | Нет orphan-scan для MinIO | Cleanup по retention или delete-cascade — best-effort; возможны "осиротевшие" объекты в MinIO. | Низкий: накопление < 1% данных в год. | low | Когда disk usage начнёт расти быстрее ожидаемого. |
| **TD-005** | UI отправки не поддерживает аттачи | Таблица `sent_attachments` зарезервирована, но в compose-форме нет upload-поля. | Средний: пользователи могут хотеть прикреплять файлы. | medium | Sprint 2+ (если будет запрос пользователей). |
| **TD-006** | Single worker — single point of failure | При падении worker'а sync останавливается. Restart=always покрывает crash, но не bug-loop. | Низкий: scope; ручной recovery приемлем. | low | Если масштабирование потребует двух+ worker — отдельный ADR с координацией задач. |
| **TD-007** | Нет автоматического failover Postgres | Single-host. При падении хоста — manual recovery из backup. | Средний: RTO ~ часы. | medium | Когда требования к доступности повысятся. |
| **TD-008** | Нет CAPTCHA на login | rate-limit + lockout есть, но distributed brute-force с пула IP теоретически возможен. | Низкий: закрытый сервис, фиксированный набор пользователей. | low | Если станет публичным или появится реальный риск. |
| **TD-010** | Нет сохранения IMAP-флагов (read/seen back-sync) | UI mark-read локальный, не синхронизируется обратно в IMAP. | Средний: пользователи могут ожидать sync с другими клиентами. | medium | Если придёт явный фидбэк. |
| **TD-011** | `apply_to_existing` для тегов — синхронный, лимит 100k messages | См. ADR-0017 §7. На текущем масштабе (≤30k messages/user) синхронное применение приемлемо (<5 сек), но при росте упрётся в HTTP-таймаут 30 сек. Лимит 100k защитит от висящих запросов. | Низкий на старте, средний при росте. | low | При росте до >50k messages у любого user'а — переделать на background worker job + UI status polling. |
| ~~**TD-012**~~ | ~~Pattern в tag_rules не escape'ит `%`/`_` (ILIKE wildcards)~~ | **Снят round-23 (TD-021)**: `*_contains` больше не использует ILIKE. Pattern теперь экранируется через `regexp_replace` и матчится whole-word через `~` + `\y`. `%`/`_` интерпретируются буквально (как обычные символы), wildcard-побочный-эффект исчез. | — | — | — |
| **TD-021** | `*_contains` теги — семантика изменена с substring-case-insensitive на whole-word **case-SENSITIVE** | **round-23 (фикс ложных срабатываний).** ADR-0017 §4 описывал substring-ILIKE (case-insensitive); паттерн `PLA` ложно цеплялся к «ex**pla**ining», «tem**pla**te», «dis**pla**y». В `backend/app/tags/sql.py` (`APPLY_TAGS_TO_MESSAGE`, `APPLY_TAG_TO_EXISTING`) заменено на whole-word **case-sensitive**: `value ~ ('\y' \|\| regexp_replace(r.pattern, '([\^$.\|?*+()\[\]{}\\])', '\\\1', 'g') \|\| '\y')` (оператор `~`, НЕ `~*`). **Регистр контролирует пользователь самим паттерном**: правило `DPLA` (капс) ловит только капсовое слово `DPLA` (напр. `Program Licence Agreement ("DPLA")`), а `dpla`/`pla` — нет. Двойная защита от ложных срабатываний: неверный регистр (`pla` ≠ `PLA`) **и** подстрока внутри слова (`\y`). Метасимволы паттерна экранируются → литеральный матч (точка в `DPLA.PLA` — литерал, не wildcard). `sender_exact` без изменений (`LOWER(...)=LOWER(...)` — email/домены де-факто регистронезависимы, сознательно остаётся case-insensitive). ReDoS-инвариант ADR-0017 §4/A2 сохранён (структура regex от пользователя не доходит до движка — всё экранируется). | Семантический: паттерны-подстроки (часть слова) и паттерны в другом регистре перестанут срабатывать — это и есть желаемое поведение. | medium | **architect должен формализовать**: обновить ADR-0017 §4/§5/§7 (substring-case-insensitive → whole-word case-sensitive) и UI-примечание (убрать «`%`/`_` поддерживаются как wildcard»; добавить «совпадение регистрозависимое»). |
| ~~**TD-013**~~ | ~~Нет push-уведомлений в Telegram о новых письмах~~ | **Закрыт ADR-0022**. Реализован полный flow: линковка через `telegram_links` + диспатчер `worker.tg_notify_dispatch` + opt-out через `users_settings.tg_notifications_enabled` + idempotency через `telegram_notifications`. | — | — | — |
| ~~**TD-014**~~ | ~~Имя env-var для bot-токена расходится между кодом и docs (`BOT_TOKEN` vs `TELEGRAM_BOT_TOKEN`)~~ | **Закрывается одновременно с ADR-0022 migration**: `shared/config.py` переименовывает `BOT_TOKEN` → `TELEGRAM_BOT_TOKEN` (alias на переходный период), `.env.example` синхронизируется. Devops применяет на prod в том же спринте. | — | — | — |
| **TD-018** | Нет anti-replay set'а для `init_data` в SSO | См. ADR-0022 Open question Q-001-1 и §1.4 «Replay украденного init_data». Текущая защита — TTL 5 минут. Внутри окна, теоретически, replay возможен (требует MitM между Telegram-edge и backend). Anti-replay set (`tg_seen:{auth_date}:{hash[:8]}` в Redis TTL 5min) сверху TTL — отложен. | Низкий: scope (≤5 users); требует cooperating MitM, что не realistic для нашего деплоймента (TLS до nginx). | low | Если появится реальный риск (например, прокси-логика leak'нула токен) — добавить SET-NX перед HMAC-валидацией в `TelegramAuthService.validate_init_data`. |
| **TD-015** | Per-group tags (теги общие для группы) | См. ADR-0019 §11 + §7.4. На текущей итерации теги остаются per-user (ADR-0017). В UX группы это даёт quirk: лидер видит письма участника со «своими» тегами не применёнными (теги — атрибут владельца ящика, не зрителя). | Средний: непривычный UX в режиме группы. Функционально не блокирует. | medium | После валидации UX модели групп: либо `tags.group_id NULLABLE` (общие теги для группы), либо «копировать теги лидера всем участникам» при добавлении в группу. Отдельный ADR. |
| **TD-016** | i18n framework для UI | См. ADR-0021. Сейчас весь UI на русском с error-code mapping в Jinja-macro `error_text(code)`. Если когда-то понадобится EN-вариант (выход на международный рынок) — придётся ретроактивно поднимать gettext/babel и переписывать ~15 шаблонов. | Низкий: продукт сейчас однозначно RU-only, единственный язык. | low | При появлении явного запроса на EN/ES/etc — отдельный ADR с миграцией на gettext/babel + extraction script + CI-step. |
| **TD-017** | Назначение лидера в существующей пустой группе | См. ADR-0019 §5 + §11. Endpoint `PATCH /api/admin/groups/{id}/leader` (либо аналогичный flow в `PATCH /api/admin/users`) не реализован. На сегодня «новый лидер всегда auto-create'ит группу»; перенос лидерства между группами — через delete+create. | Низкий: продукт явно не запросил такой flow. На малом масштабе (≤3 групп) приемлемо. | low | При появлении сценария «передать лидерство без удаления группы» — отдельный ADR + endpoint с revoke_all_for_user старого лидера + audit `user_role_change`. |
| **TD-019** | Outbound webhook rotate-secret — instant cut без grace-периода | См. ADR-0023 Open question Q-WH-1 и §2.5. После `POST /api/webhooks/me/rotate-secret` старый secret становится недействителен немедленно — receiver-side приложение, не успевшее принять новый secret, видит unauth-error до момента deploy с новым env. Двойного secret (старый ещё валиден M минут) в MVP нет. | Низкий: ротация редкая операция, обычно координируется ручно (сообщение «обновите secret»). На малом масштабе (≤5 команд) приемлемо. | low | Если поступит запрос с обоснованием «нужен hot-rotate без downtime» — добавить колонку `webhooks.secret_previous_encrypted` + `secret_previous_valid_until: timestamptz`; dispatcher шлёт оба header'а (`X-Webhook-Secret` + `X-Webhook-Secret-Previous`) на промежуточный период. Отдельный ADR (`ADR-0023.1`). |
