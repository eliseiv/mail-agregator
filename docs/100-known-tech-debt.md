| ~~**TD-023**~~ | ~~Границы слова через `\y` ломают паттерны, обрамлённые пунктуацией~~ | **Закрыт round-27.** `\y` = граница «словесный↔несловесный»; паттерн, начинающийся/заканчивающийся пунктуацией (`Congratulations!`, `…requires your attention.`), не матчился — после `!`/`.` идёт пробел/конец (обе стороны несловесные, границы нет). Ломало теги «Реджект»/«Релиз» (особенно `match_mode='all'`, где один несработавший rule снимает весь тег). В `backend/app/tags/sql.py` `\y…\y` заменено на явные граничные классы `'(^|[^[:alnum:]_])' \|\| <escaped> \|\| '([^[:alnum:]_]\|$)'` во всех 8 местах (2 запроса × any/all × subject/body/sender). Гарантии whole-word/case-sensitive сохранены. Формализовано в ADR-0017 §4.1. | Функциональный баг (теги не навешивались) — устранён. | — | — |
| **TD-022** | `norm()` + `strip_tags(body_html)` выполняются на каждой проверке rule без индекса | round-27: чтобы многословные `*_contains` матчились на реальных телах (где `html2text`/`text-plain` дают `\n`, прогоны пробелов, U+00A0 внутри предложений — замер: 148/148 писем с `\n`, 108/148 с прогонами, 17/148 с nbsp), обе стороны сравнения проходят `norm(x)=regexp_replace(translate(x,chr(160),' '),'\s+',' ','g')`. **round-29:** `body_contains` теперь дополнительно прогоняет `strip_tags(body_html)=regexp_replace(body_html,'<[^>]+>',' ','g')` (до ~1 MiB на строку) → на full-scan apply-to-existing per-row CPU тела удваивается. Это per-row CPU без функционального индекса по нормализованному `body_text`/`body_html`. См. ADR-0017 §4.2/§4.3. | Низкий: на ≤750k писем и batch ≤50 укладывается в sync-window; runaway-guard 100k и seq-проверка по `tag_rules` доминируют. | low | При росте до >50k писем/user с заметным временем apply — материализованная колонка `messages.body_searchable` (`body_text` + `strip_tags(body_html)`, + trigger) либо expression-индекс под GIN/pg_trgm. Совместить с пересмотром TD-011 (async apply). |
| **TD-024** | `strip_tags(body_html)` не декодирует HTML-сущности | round-29 (ADR-0017 §4.3): `body_contains` матчит дополнительно по `strip_tags(body_html)=regexp_replace(body_html,'<[^>]+>',' ','g')` — снимает только теги `<…>`, но **не** декодирует сущности (`&amp;`→`&`, `&#39;`→`'`, `&nbsp;`→U+00A0). Паттерн, совпадающий с фразой, где в html стоит сущность (напр. паттерн `AT&T` против html `AT&amp;T`), через html-ветку не сматчится. `&nbsp;` частично смягчён `norm()` — но только если сущность уже декодирована в U+00A0; сырой `&nbsp;` остаётся литералом. Для подтверждённого Apple-кейса (письма id 265/381) фраза чистая — фикс работает. | Низкий: типовые триггерные фразы Apple/App Store без сущностей; затрагивает только html-ветку (body_text-ветка не страдает). | low | Если появятся пользовательские паттерны с `&`/кавычками/nbsp, не матчащиеся из-за сущностей — добавить unescape в SQL (цепочка `replace` для топ-сущностей `&amp;`/`&lt;`/`&gt;`/`&quot;`/`&#39;`/`&nbsp;`) или вынести strip+unescape в `html2text` на стороне воркера в материализованную колонку (см. TD-022). |

Follow-up «вынести visibility в общий фрагмент» выполнен. | — | — | — |
| **TD-026** | Глобальный Bot API rate-limit (~30 msg/sec на всех чатах) не throttled явно | round-31 (ADR-0022 §2.9): per-chat троттлинг (`TG_SEND_PER_CHAT_PER_MINUTE`) ограничивает поток в один chat_id, но глобального лимита на суммарный исходящий поток бота нет. Смягчено `TG_NOTIFY_BATCH_SIZE=30`/тик при `TG_NOTIFY_DISPATCH_INTERVAL=5s` (~6 msg/sec глобально, ниже 30/sec) + backoff на `429` (rollback + re-enqueue). | Низкий: текущая batch/interval-конфигурация держит глобальный поток заведомо ниже лимита Bot API; всплеск при `TG_NOTIFY_ALL_MESSAGES=true` + много активных чатов теоретически возможен, но `429` обрабатывается re-enqueue без потери. | low | Если на проде будут стабильные `429` — добавить глобальный token-bucket (`rl:tg_send_global`, ~25/sec) проверкой в диспатчере перед каждым `send_notification`. |
| **TD-027** | При **устойчивом** per-chat throttle часть TG-нотификаций может не доставиться в окне 24ч | round-32 (заведён), **переформулирован round-33** (ADR-0022 §2.9). Причина недоставки — **устойчивый throttle** (`inflow > TG_SEND_PER_CHAT_PER_MINUTE` дольше окна), а НЕ маскированная частичная доставка (последнее было отдельным CRITICAL-багом per-message recovery, исправлено round-33 — см. §2.8, TD-025). Механизм: throttled-получатель пропускается без hot re-enqueue (`continue`, строки `(message_id,user_id)` нет); доставку берёт per-recipient `recovery_scan` (раз в час, окно `TG_NOTIFY_RECOVERY_WINDOW_HOURS=24`). При устойчивом флуде backlog рассасывается ~capacity попыток/чат/час; письмо, не успевшее пройти throttle до выхода за 24ч, recovery перестаёт подбирать → TG-нотификация по нему не доставится. Само письмо НЕ теряется (в БД, видно в UI) — теряется только уведомление. Осознанный компромисс: альтернатива (немедленный hot re-enqueue) давала busy-loop + неограниченный рост `tg_notify_queue` + амплификацию recipient-SQL каждые 5с. | Низкий: только при устойчивом флуде в один chat дольше суток (нетипично для ≤5 users; super_admin может выключить `tg_notifications_enabled` или откатить `TG_NOTIFY_ALL_MESSAGES`). Письмо доступно в UI. | low | Если устойчивый флуд станет реальным — поднять `TG_SEND_PER_CHAT_PER_MINUTE` и/или `TG_NOTIFY_RECOVERY_WINDOW_HOURS`, либо ввести приоритетную/persistent per-recipient очередь с явным счётчиком попыток вместо опоры на recovery-окно. |
| **TD-019** | Outbound webhook rotate-secret — instant cut без grace-периода | См. ADR-0023 Open question Q-WH-1 и §2.5. После `POST /api/webhooks/me/rotate-secret` старый secret становится недействителен немедленно — receiver-side приложение, не успевшее принять новый secret, видит unauth-error до момента deploy с новым env. Двойного secret (старый ещё валиден M минут) в MVP нет. | Низкий: ротация редкая операция, обычно координируется ручно (сообщение «обновите secret»). На малом масштабе (≤5 команд) приемлемо. | low | Если поступит запрос с обоснованием «нужен hot-rotate без downtime» — добавить колонку `webhooks.secret_previous_encrypted` + `secret_previous_valid_until: timestamptz`; dispatcher шлёт оба header'а (`X-Webhook-Secret` + `X-Webhook-Secret-Previous`) на промежуточный период. Отдельный ADR (`ADR-0023.1`). |
